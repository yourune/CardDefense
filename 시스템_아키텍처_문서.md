# CardDefense 시스템 아키텍처 문서

## 1. 전체 아키텍처 개요

### 1.1 핵심 설계 철학

CardDefense는 **이벤트 기반 액션 시스템(Event-Driven Action System)**을 중심으로 설계된 카드 디펜스 게임입니다. 모든 게임 로직은 `GameAction`을 통해 실행되며, 각 액션은 3단계 생명주기(PRE → PERFORM → POST)를 거치면서 다른 액션을 반응(Reaction)으로 생성할 수 있습니다.

### 1.2 아키텍처 레이어 구조

```
┌─────────────────────────────────────────┐
│         Presentation Layer              │
│   (View, UI, Creators)                  │
├─────────────────────────────────────────┤
│         Application Layer               │
│   (Systems - 비즈니스 로직)              │
├─────────────────────────────────────────┤
│         Domain Layer                    │
│   (Models, Effects, TargetModes)        │
├─────────────────────────────────────────┤
│         Infrastructure Layer            │
│   (ActionSystem, GameActions)           │
├─────────────────────────────────────────┤
│         Data Layer                      │
│   (ScriptableObject Data)               │
└─────────────────────────────────────────┘
```

### 1.3 핵심 디자인 패턴

- **ActionSystem Pattern**: 모든 게임 상태 변경을 GameAction으로 캡슐화
- **Singleton Pattern**: 각 시스템의 전역 접근성 보장
- **Factory Pattern**: View 객체 생성 관리
- **Strategy Pattern**: Effect와 TargetMode의 다형성 구현
- **Command Pattern**: GameAction이 커맨드 역할 수행
- **Observer Pattern**: ActionSystem의 구독-발행 메커니즘

## 2. Infrastructure Layer - ActionSystem 핵심 시스템

### 2.1 ActionSystem - 게임의 심장부

`ActionSystem`은 전체 게임의 상태 관리를 담당하는 핵심 시스템입니다. Unity의 싱글톤으로 구현되어 있으며, 모든 게임 액션의 실행을 조율합니다.

#### 핵심 메커니즘

```
GameAction 실행 흐름:
┌─────────────────────────────────────────────┐
│  1. PRE Phase                               │
│     - PreReactions 수집                     │
│     - Pre Subscribers 실행                  │
│     - PreReactions의 모든 액션 재귀 실행    │
├─────────────────────────────────────────────┤
│  2. PERFORM Phase                           │
│     - PerformReactions 수집                 │
│     - 등록된 Performer 실행 (핵심 로직)     │
│     - PerformReactions의 모든 액션 재귀 실행│
├─────────────────────────────────────────────┤
│  3. POST Phase                              │
│     - PostReactions 수집                    │
│     - Post Subscribers 실행                 │
│     - PostReactions의 모든 액션 재귀 실행   │
└─────────────────────────────────────────────┘
```

#### 주요 API

- `Perform(GameAction action)`: 액션 실행 시작점
- `AddReaction(GameAction action)`: 현재 실행 중인 액션에 반응 추가
- `AttachPerformer<T>(Func<T, IEnumerator> performer)`: 시스템이 특정 GameAction 타입 처리 등록
- `DetachPerformer<T>()`: Performer 등록 해제
- `SubscribeReaction<T>(Action<T>, ReactionTiming)`: 액션 발생 시 리스너 등록
- `UnsubscribeReaction<T>(Action<T>, ReactionTiming)`: 리스너 해제

#### 설계 특징

1. **재귀적 액션 체인**: 각 액션이 다른 액션을 Reaction으로 추가할 수 있어 복잡한 게임 로직을 간단하게 표현
2. **큐 기반 실행**: 여러 액션이 동시에 요청되면 큐에 저장하고 순차적으로 실행하여 안정성 보장
3. **타입 기반 라우팅**: 제네릭과 Dictionary를 활용한 효율적인 액션 디스패칭
4. **코루틴 기반**: Unity의 비동기 처리를 활용한 애니메이션 통합

#### 큐 시스템

ActionSystem은 내부적으로 `Queue<(GameAction, Action)>`를 사용하여 액션 대기열을 관리합니다:

```csharp
// 액션 요청 시
public void Perform(GameAction action, Action OnPerformFinished = null)
{
    actionQueue.Enqueue((action, OnPerformFinished));
    
    if (!isProcessingQueue)
    {
        StartCoroutine(ProcessQueue());
    }
}

// 큐 처리
private IEnumerator ProcessQueue()
{
    isProcessingQueue = true;
    
    while (actionQueue.Count > 0)
    {
        var (action, callback) = actionQueue.Dequeue();
        yield return Flow(action);
        callback?.Invoke();
    }
    
    isProcessingQueue = false;
}
```

**장점**:
- 여러 시스템에서 동시에 액션을 호출해도 안전
- 액션 실행 순서 보장
- 콜백을 통한 개별 액션 완료 알림
- `IsPerforming` 프로퍼티로 현재 상태 확인 가능

### 2.2 GameAction - 모든 게임 이벤트의 기반

```csharp
public abstract class GameAction
{
    public List<GameAction> PreReactions;
    public List<GameAction> PerformReactions;
    public List<GameAction> PostReactions;
}
```

#### GameAction의 역할

- **데이터 캡슐화**: 액션 실행에 필요한 모든 정보 포함
- **불변성(Immutability)**: 한 번 생성된 액션은 수정되지 않음 (대부분 private set)
- **재귀 트리거 구조**: Reactions 리스트를 통해 연쇄 액션 생성

#### 구현된 GameAction 종류

##### 카드 관련
- `PlayCardGA`: 카드 플레이 (마나 소비, 효과 발동 포함)
- `DrawCardsGA`: 카드 드로우
- `DiscardAllCardsGA`: 모든 카드 버리기

##### 효과/전투 관련
- `DealDamageGA`: 데미지 처리
- `PerformEffectGA`: 효과 실행 래퍼
- `KillEnemyGA`: 적 제거
- `AttackCastleGA`: 성 공격

##### 자원 관련
- `SpendManaGA`: 마나 소비
- `RefillManaGA`: 마나 회복
- `GainGoldGA`: 골드 획득
- `GainXpGA`: 경험치 획득
- `LevelUpGA`: 레벨 업 처리
- `DropRewardGA`: 보상 시각 효과 트리거

### 2.3 Singleton Pattern 구현

```csharp
public abstract class Singleton<T> : MonoBehaviour where T : MonoBehaviour
public abstract class PersistentSingleton<T> : Singleton<T> where T : MonoBehaviour
```

- **Singleton**: 씬 전환 시 파괴되는 싱글톤
- **PersistentSingleton**: DontDestroyOnLoad로 씬 전환 후에도 유지

모든 주요 시스템은 Singleton을 상속하여 `[SystemName].Instance`로 전역 접근 가능합니다.

## 3. Data Layer - 게임 데이터 정의

### 3.1 ScriptableObject 기반 데이터

Unity의 ScriptableObject를 활용한 데이터 중심 설계로, 기획자가 에디터에서 쉽게 수정 가능합니다.

#### CardData
```csharp
- Description: 카드 설명 텍스트
- Mana: 마나 비용
- Image: 카드 이미지
- CastImage: 캐스팅 시 커서 이미지
- ManualTargetEffect: 플레이어가 직접 타겟팅하는 효과 (nullable)
- OtherEffects: 자동 타겟팅 효과 리스트
```

#### CastleData
```csharp
- Image: 성 이미지
- Health: 체력
- Deck: 덱 구성 (CardData 리스트)
```

#### EnemyData
```csharp
- Image: 적 이미지
- Health: 체력
- Speed: 이동 속도
- GoldDrop: 처치 시 드랍하는 골드량 (기본: 10)
- XpDrop: 처치 시 드랍하는 경험치량 (기본: 5)
```

### 3.2 데이터 → 모델 → 뷰 흐름

```
ScriptableObject (CardData)
    ↓
Model (Cards) - 런타임 데이터 래핑
    ↓
View (CardView) - 화면 표시
```

이 구조는 **데이터와 로직의 분리**, **재사용성**, **테스트 용이성**을 제공합니다.

## 4. Domain Layer - 게임 로직 추상화

### 4.1 Cards Model

`Cards` 클래스는 `CardData`의 런타임 래퍼입니다. 실행 중 변경 가능한 상태(예: 마나 비용)를 관리합니다.

```csharp
public class Cards
{
    private readonly CardData data;  // 불변 데이터 참조
    public int Mana { get; private set; }  // 런타임에 변경 가능
}
```

### 4.2 Effect System - Strategy Pattern

`Effect`는 카드의 실제 효과를 정의하는 추상 클래스입니다.

```csharp
public abstract class Effect
{
    public abstract GameAction GetGameAction(List<CombatantView> targets);
}
```

#### 구현된 Effect 종류

- `DealDamageEffect`: 데미지 효과 (damageAmount 파라미터)
- `DrawCardsEffect`: 카드 드로우 효과 (drawAmount 파라미터)

#### 확장성

새로운 효과 추가 방법:
1. `Effect` 클래스 상속
2. `GetGameAction` 메서드에서 적절한 `GameAction` 반환
3. 에디터에서 `[System.Serializable]` 속성으로 인스펙터 노출

### 4.3 TargetMode System - Strategy Pattern

`TargetMode`는 효과의 타겟 선택 로직을 정의합니다.

#### 구현된 TargetMode 종류

- `NoTM`: 타겟 없음 (null 반환)
- `RandomEnemyTM`: 랜덤 적 1명
- `AllEnemiesTM`: 모든 적

### 4.4 AutoTargetEffect - Effect와 TargetMode의 조합

```csharp
public class AutoTargetEffect
{
    public TargetMode TargetMode;
    public Effect Effect;
}
```

카드의 자동 효과는 "누구에게(TargetMode) + 무엇을(Effect)" 조합으로 표현됩니다.

## 5. Application Layer - Systems

### 5.1 CardSystem - 카드 관리의 중추

카드의 생명주기 전체를 관리하는 시스템입니다.

#### 관리 대상
- `drawPile`: 드로우 덱
- `discardPile`: 버린 카드 더미
- `hand`: 현재 손패

#### 핵심 로직

**DrawCardsGA 처리**
```
1. 요청된 수만큼 드로우 시도
2. 덱이 부족하면 discardPile을 drawPile로 리필
3. HandView에 카드 추가
4. 손패 제한(maxHandSize=10) 초과 시 왼쪽부터 자동 버림
```

**PlayCardGA 처리**
```
1. 손패에서 카드 제거
2. 카드뷰를 discardPile로 이동 (애니메이션)
3. SpendManaGA를 Reaction으로 추가
4. ManualTargetEffect가 있으면 PerformEffectGA 추가
5. OtherEffects 순회하며 각각 PerformEffectGA 추가
```

**DiscardAllCardsGA 처리**
- 손패의 모든 카드를 버림 더미로 이동

### 5.2 ManaSystem - 자원 관리

마나는 게임의 핵심 자원으로, 다음과 같이 관리됩니다:

- **MAX_MANA**: 10
- **STARTING_MANA**: 5
- **자동 회복**: 3초마다 1씩 자동 회복 (AutoRefillMana 코루틴)

#### API
- `HasEnoughMana(int amount)`: 마나 소지 여부 확인 (카드 플레이 전 검증)
- `SpendManaGA` Performer: 마나 차감 및 UI 업데이트
- `RefillManaGA` Performer: 마나 회복 및 UI 업데이트

### 5.3 EnemySystem - 적 생명주기 관리

#### 책임
1. **적 스폰**: `SpawnEnemy(List<EnemyData>)` - 일정 간격으로 적 생성
2. **적 이동**: `Update`에서 모든 활성 적을 왼쪽으로 이동
3. **성 도달 감지**: x좌표 비교로 성 도달 확인 → `AttackCastleGA` 발동
4. **적 사망 처리**: `KillEnemyGA` 처리

#### 중요 메커니즘

**AttackCastleGA Performer**
```csharp
DealDamageGA dealDamageGA = new DealDamageGA(damage, targets);
ActionSystem.Instance.AddReaction(dealDamageGA);
```
공격을 바로 처리하지 않고 `DealDamageGA`를 Reaction으로 추가하여, DamageSystem이 일관되게 처리하도록 위임합니다.

### 5.4 DamageSystem - 데미지 계산 및 사망 처리

모든 데미지는 이 시스템을 통해 처리됩니다.

#### DealDamageGA Performer 흐름
```
1. 각 타겟에 대해 순회
2. target.TakeDamage(damage) 호출
3. 데미지 VFX 생성
4. 체력이 0 이하인지 확인
   - EnemyView인 경우: KillEnemyGA를 Reaction으로 추가
   - CastleView인 경우: 게임 오버 로직 (미구현)
```

이 구조의 장점:
- 데미지 처리 로직이 한 곳에 집중
- 사망 처리를 별도 액션(KillEnemyGA)으로 분리하여 다른 시스템이 후처리 가능

### 5.5 EffectSystem - 효과 실행 중재자

`EffectSystem`은 매우 간단하지만 중요한 역할을 합니다:

```csharp
private IEnumerator PerformEffectPerformer(PerformEffectGA performEffectGa)
{
    GameAction effectAction = performEffectGa.Effect.GetGameAction(performEffectGa.Targets);
    ActionSystem.Instance.AddReaction(effectAction);
    yield return null;
}
```

`Effect` 객체에서 `GameAction`을 생성하고, 이를 Reaction으로 추가합니다. 이를 통해 Effect와 실제 GameAction 실행을 분리합니다.

### 5.6 CastleSystem - 성 관리

가장 단순한 시스템으로, `CastleView`의 초기 설정만 담당합니다.

```csharp
public CastleView CastleView { get; private set; }
public void Setup(CastleData castleData)
```

### 5.7 Interactions System - 플레이어 입력 상태 관리

전역 상태 관리자로, 플레이어가 게임과 상호작용 가능한지 판단합니다.

```csharp
- PlayerIsDragging: 카드를 드래그 중인지 여부
- PlayerCanInteract(): ActionSystem이 실행 중이 아닐 때만 true
- PlayerCanHover(): 드래그 중이 아닐 때만 true
```

다른 시스템들이 입력을 받기 전 `Interactions.Instance.PlayerCanInteract()`로 검증합니다.

### 5.8 ManualTargetSystem - 플레이어 타겟팅

플레이어가 적을 직접 타겟팅할 때의 상태를 관리합니다.

#### 상태 흐름
```
StartTargeting(카드 이미지들)
    ↓
UpdateTargeting(마우스 위치) - 매 프레임
    ↓ (Raycast로 적 탐지)
    ↓ (타겟 인디케이터 on/off)
    ↓ (커서 이미지 변경)
    ↓
EndTargeting(최종 위치) → EnemyView 또는 null 반환
```

### 5.9 ChangeCursorSystem - 동적 커서 관리

게임 중 커서를 카드 이미지나 캐스팅 이미지로 변경하는 시스템입니다.

#### 기능
- `ShowCardImage(Sprite)`: 일반 카드 이미지로 커서 변경
- `ShowCastImage(Sprite)`: 캐스팅 이미지로 커서 변경 (타겟 위에 올릴 때)
- `HideCursorImage()`: 원래 커서로 복원
- `IsOverDropArea()`: 카드를 떨어뜨릴 수 있는 영역인지 Raycast 검사

커서 이미지는 UI Image로 구현되어 마우스 위치를 따라다닙니다.

### 5.10 CardViewHoverSystem - 카드 프리뷰

손패의 카드 위에 마우스를 올렸을 때 확대 프리뷰를 표시합니다.

```csharp
Show(Cards cards, Vector3 position)  // 프리뷰 표시
Hide()  // 프리뷰 숨김
```

### 5.11 AutoDrawSystem - 자동 카드 드로우

```csharp
private IEnumerator AutoDrawCards()
{
    while (true)
    {
        yield return new WaitForSeconds(autoDrawInterval);  // 5초
        DrawCardsGA drawCardsGA = new(1);
        ActionSystem.Instance.Perform(drawCardsGA);
    }
}
```

일정 간격으로 자동으로 카드를 드로우하는 간단한 게임 메커니즘입니다.

### 5.12 MatchSetupSystem - 게임 초기화

게임 시작 시 모든 시스템을 초기화하는 진입점입니다.

```csharp
void Start()
{
    CastleSystem.Instance.Setup(CastleData);
    EnemySystem.Instance.SpawnEnemy(EnemyDataList);
    CardSystem.Instance.SetUp(CastleData.Deck);
    DrawCardsGA drawCardsGA = new(5);  // 초기 5장 드로우
    ActionSystem.Instance.Perform(drawCardsGA);
}
```

### 5.13 RewardSystem - 보상 및 레벨링 관리

골드, 경험치, 레벨 시스템을 관리하는 시스템입니다.

#### 관리 대상
- `CurrentGold`: 현재 보유 골드
- `CurrentXp`: 현재 경험치
- `CurrentLevel`: 현재 레벨
- `XpRequiredForNextLevel`: 다음 레벨까지 필요한 경험치

#### 핵심 로직

**레벨 진행 공식**
```
Level 2: 100 XP
Level 3: 150 XP (100 × 1.5)
Level 4: 225 XP (150 × 1.5)
XpRequired = baseXpRequired × (scalingFactor ^ (level - 1))
```

**GainXpGA 처리**
```
1. 경험치 추가
2. 레벨업 조건 확인 (currentXp >= required)
3. 조건 충족 시 LevelUpGA를 Reaction으로 추가
4. UI 업데이트 이벤트 발생
```

**LevelUpGA 처리**
```
1. 레벨 증가
2. 다음 레벨 경험치 재계산
3. 성의 최대 체력 증가 (기본 +10)
4. UI 업데이트 및 레벨업 이벤트 발생
```

#### 이벤트 시스템
```csharp
public event Action<int> OnGoldChanged;
public event Action<int, int, int> OnXpChanged; // currentXp, required, level
public event Action<int> OnLevelUp;
```

다른 시스템이나 UI가 구독하여 실시간으로 변경사항을 반영할 수 있습니다.

### 5.14 RewardVisualSystem - 보상 시각 효과

적이 죽을 때 골드와 경험치 드랍의 시각적 피드백을 제공합니다. **오브젝트 풀링**을 사용하여 성능을 최적화했습니다.

#### DropRewardGA 처리 흐름
```
1. 적의 위치에서 플로팅 텍스트 생성 ("+10G +5XP")
2. 골드 코인 스폰 및 애니메이션 (풀에서 가져옴)
   - 생성 개수: Clamp((goldAmount / 20) + 1, 1, 3)
   - 위로 튕기는 드랍 애니메이션
   - UI 골드 아이콘으로 이동
   - 수집 애니메이션
   - 애니메이션 완료 후 풀로 반환
3. 경험치 오브 스폰 및 애니메이션 (풀에서 가져옴)
   - 생성 개수: Clamp((xpAmount / 10) + 1, 1, 3)
   - 위로 튕기는 드랍 애니메이션
   - UI 경험치 바로 이동
   - 수집 애니메이션
   - 애니메이션 완료 후 풀로 반환
4. 애니메이션 완료 후 실제 자원 획득
   - GainGoldGA 실행
   - GainXpGA 실행
```

#### 오브젝트 풀링 시스템
- **SimpleObjectPool\<T\>**: 제네릭 풀링 시스템
  - 시작 시 `initialPoolSize`(기본 20개)만큼 미리 생성
  - `Get()`: 풀에서 객체 가져오기 (없으면 새로 생성)
  - `Return()`: 사용 완료 후 풀로 반환
  - 자동 확장: 풀이 비면 자동으로 새 객체 생성
- **장점**:
  - Instantiate/Destroy 호출 최소화 → 성능 향상
  - GC 압력 감소 → 프레임 드랍 방지
  - 대규모 전투에서도 안정적인 성능

#### 생성 개수 최적화
- **이전**: 10골드당 1개, 최대 5개 (많이 생성됨)
- **현재**: 20골드당 1개, 최대 3개 (깔끔하고 성능 좋음)
- **최소 보장**: 항상 최소 1개는 생성됨

#### 애니메이션 파라미터
- `dropDelay`: 코인/오브 간 딜레이 (0.1초)
- `dropDuration`: 드랍 애니메이션 시간 (0.8초)
- `dropHeight`: 드랍 높이 (2 유닛)
- `collectDuration`: 수집 애니메이션 시간 (0.5초)

시각적 피드백 → 실제 자원 획득 순서를 통해 플레이어에게 보상의 즉각적인 피드백을 제공합니다.

## 9. 핵심 설계 원칙 및 패턴 요약

### 9.1 SOLID 원칙 적용

#### Single Responsibility Principle (단일 책임 원칙)
- 각 System은 하나의 도메인만 담당 (CardSystem = 카드, ManaSystem = 마나)
- GameAction은 데이터만 담고, 실제 로직은 System의 Performer가 처리

#### Open/Closed Principle (개방-폐쇄 원칙)
- Effect, TargetMode는 새 클래스 추가로 확장 가능, 기존 코드 수정 불필요
- ActionSystem은 제네릭을 통해 새 GameAction 타입을 자동 지원

#### Liskov Substitution Principle (리스코프 치환 원칙)
- CombatantView의 모든 서브클래스(EnemyView, CastleView)는 교체 가능
- GameAction의 모든 서브클래스는 동일한 Flow 메커니즘으로 처리됨

#### Interface Segregation Principle (인터페이스 분리 원칙)
- Effect와 TargetMode는 작은 추상 클래스로 분리
- 각 시스템은 필요한 메서드만 public으로 노출

#### Dependency Inversion Principle (의존성 역전 원칙)
- 상위 레벨(Systems)은 하위 레벨(View) 구체 클래스에 직접 의존하지 않음
- ActionSystem을 통한 간접 통신으로 결합도 감소

### 9.2 주요 디자인 패턴 상세

#### Command Pattern (GameAction)
- 각 GameAction은 실행 가능한 커맨드 객체
- Undo/Redo는 미구현이지만 구조적으로 가능
- 커맨드 체인(Reactions)을 통한 복잡한 로직 구성

#### Observer Pattern (ActionSystem Subscribers)
- SubscribeReaction으로 특정 액션에 리스너 등록
- 이벤트 기반 시스템 간 통신
- 느슨한 결합 유지

#### Strategy Pattern (Effect, TargetMode)
- 런타임에 효과와 타겟팅 전략 교체 가능
- 카드 데이터에서 전략 주입(SerializeReference)
- 알고리즘군을 캡슐화

#### Factory Pattern (Creators)
- 복잡한 View 객체 생성 로직 캡슐화
- 일관된 초기화 보장
- 프리팹 의존성 중앙 관리

#### Singleton Pattern (Systems)
- 전역 접근점 제공
- 생명주기 관리 (Persistent 여부 선택 가능)
- 게임 내 유일성 보장

### 9.3 ActionSystem의 설계 의도

#### 왜 Reaction 시스템을 사용하는가?

1. **순서 보장**: 마나 소비 → 효과 발동 → 사망 처리 순서가 명확함
2. **확장성**: 새로운 효과가 또 다른 효과를 트리거 가능 (연쇄 반응)
3. **디버깅 용이성**: 각 액션이 명시적이어서 로그 추적이 쉬움
4. **애니메이션 통합**: 코루틴으로 비동기 애니메이션을 동기적 흐름에 통합
5. **모듈화**: 시스템 간 직접 호출 없이 Reaction으로 통신

#### 3단계 생명주기 (PRE/PERFORM/POST)의 의도

- **PRE**: "~하기 전에" 로직 (예: 카드 플레이 전 버프 적용)
- **PERFORM**: 핵심 로직
- **POST**: "~한 후에" 로직 (예: 카드 사용 후 트리거)

TCG(Trading Card Game)의 복잡한 타이밍 규칙을 코드로 표현하기 위한 구조입니다.

## 10. 새로운 기능 추가 가이드라인

### 10.1 새로운 카드 효과 추가

#### 예시: "적에게 3 데미지를 주고, 카드 1장을 드로우하는 효과"

**1단계: Effect 클래스 생성**
```csharp
[System.Serializable]
public class DamageAndDrawEffect : Effect
{
    [SerializeField] private int damageAmount = 3;
    [SerializeField] private int drawAmount = 1;
    
    public override GameAction GetGameAction(List<CombatantView> targets)
    {
        // 이미 존재하는 GameAction 재사용
        DealDamageGA damageGA = new DealDamageGA(damageAmount, targets);
        
        // DrawCards는 타겟이 필요 없으므로 Reaction으로 추가
        DrawCardsGA drawGA = new DrawCardsGA(drawAmount);
        damageGA.PerformReactions.Add(drawGA);
        
        return damageGA;
    }
}
```

**2단계: 카드 데이터 설정**
- Unity 에디터에서 CardData의 OtherEffects에 추가

### 10.2 새로운 TargetMode 추가

```csharp
public class BuffSystem : Singleton<BuffSystem>
{

    void OnEnable()
    {
        ActionSystem.AttachPerformer<ApplyBuffGA>(ApplyBuffPerformer);

        ActionSystem.DetachPerformer<RemoveBuffGA>();
        ActionSystem.UnsubscribeReaction<DealDamageGA>(OnBeforeDealDamage, ReactionTiming.PRE);
    }
    
    private void OnBeforeDealDamage(DealDamageGA dealDamageGA)
    {
        // 버프에 따라 데미지 수정 (DealDamageGA는 수정 불가하므로 다른 방법 필요)
        // 실제로는 버프를 검사하고 추가 Reaction을 생성하는 방식 권장
    }
    
    private IEnumerator ApplyBuffPerformer(ApplyBuffGA action)
    {
        if (!activeBuffs.ContainsKey(action.Target))
            activeBuffs[action.Target] = new List<Buff>();
        
        activeBuffs[action.Target].Add(action.Buff);
        // 버프 VFX 생성 등
        yield return null;
    }
    
    private IEnumerator RemoveBuffPerformer(RemoveBuffGA action)
    {
        if (activeBuffs.ContainsKey(action.Target))
            activeBuffs[action.Target].Remove(action.Buff);
        yield return null;
    }
}
```

### 10.6 주의사항 및 베스트 프랙티스

#### GameAction 설계 시
1. **불변성 유지**: GameAction의 데이터는 생성 후 변경하지 않음 (private set)
2. **단일 책임**: 하나의 GameAction은 하나의 의미 있는 게임 이벤트만 표현
3. **null 처리**: 타겟이 없을 수 있는 경우 null 체크 필수

#### System 설계 시
1. **Performer 등록/해제**: OnEnable/OnDisable에서 항상 쌍으로 처리
2. **Singleton 적절성**: 정말 전역 인스턴스가 필요한지 검토
3. **ActionSystem.IsPerforming 활용**: 액션 실행 중인지 확인 필요 시 체크 (큐에 대기 중인 액션 포함)

#### Effect 설계 시
1. **재사용 가능한 GameAction 활용**: 가능한 한 새 GameAction보다 조합 사용
2. **SerializeField**: 파라미터는 에디터에서 설정 가능하도록
3. **null 안전성**: targets가 null이거나 비어있을 수 있음

#### Reaction 사용 시
1. **타이밍 고려**: PRE/PERFORM/POST 중 적절한 시점 선택
2. **무한 루프 방지**: A 액션이 B를 트리거하고 B가 다시 A를 트리거하지 않도록
3. **성능**: 과도한 Reaction 체인은 프레임 저하 유발 가능

## 12. 성능 및 아키텍처 고려사항

### 12.1 ActionSystem 큐 블로킹 문제와 해결

**문제**: ActionSystem은 단일 큐 기반으로 순차 처리하므로, 긴 작업이 전체 게임 플로우를 블록할 수 있습니다.

```csharp
// 문제 상황
PoisonZone.UpdateZone() {
    ActionSystem.Perform(DealDamageGA);  // 큐에 추가
}

ActionSystem.ProcessQueue() {
    while (queue.Count > 0) {
        yield return Flow(action);  // ← 완료될 때까지 대기
        // 이 동안 카드 플레이, 웨이브 시작 등 모든 액션 블록!
    }
}
```

**해결 원칙**: 

1. **즉시 처리 가능한 작업은 ActionSystem 우회**
   - Zone의 틱 데미지: `TakeDamage()` 직접 호출
   - 단순 상태 변경
   - 백그라운드 작업

2. **게임 상태를 변경하는 중요 작업만 ActionSystem 사용**
   - 적 죽음 (리워드, 통계 등 연계)
   - 카드 플레이 (마나, 효과 등 복잡한 로직)
   - 웨이브 관리

3. **책임 분리와 중앙 집중화**
   - **EnemySystem**: 모든 적의 죽음을 감지하고 배치 처리
   - **Zone/DamageSystem**: 데미지만 적용 (죽음 판단 X)
   - **단일 책임 원칙(SRP)** 준수

**현재 구현 예시**:

```csharp
// EnemySystem - 죽음 감지의 중앙 집중화
private void Update() {
    MoveEnemies();
    CheckAndHandleDeadEnemies();  // ← 모든 죽음을 여기서 일괄 관리
}

private void CheckAndHandleDeadEnemies() {
    List<EnemyView> deadEnemies = new List<EnemyView>();
    
    // 죽은 적 감지
    foreach (var enemy in activeEnemies) {
        if (enemy != null && enemy.currentHealth <= 0) {
            enemy.Speed = 0f;
            deadEnemies.Add(enemy);
        }
    }
    
    // 배치 처리 (한 번의 ActionSystem 호출)
    if (deadEnemies.Count > 0) {
        if (deadEnemies.Count == 1) {
            ActionSystem.Instance.Perform(new KillEnemyGA(deadEnemies[0]));
        } else {
            ActionSystem.Instance.Perform(new KillMultipleEnemiesGA(deadEnemies));
        }
    }
}

// EffectZone 베이스 클래스 - 데미지만 담당
protected void DealDamageToEnemiesInZone(int damage) {
    foreach (var enemy in enemiesInZone) {
        if (enemy != null) {
            enemy.TakeDamage(damage);  // ← 데미지만! 죽음은 EnemySystem이 처리
        }
    }
}

// PoisonZone - 극도로 간결
protected override void UpdateZone() {
    if (Time.time >= nextTickTime) {
        nextTickTime = Time.time + tickInterval;
        DealDamageToEnemiesInZone(damagePerTick);  // 끝!
    }
}
```

**장점**:
- **확장성**: 새 Zone 추가 시 죽음 처리 로직 불필요
- **일관성**: 모든 죽음이 EnemySystem을 통해 처리 → 버그 감소
- **성능**: 배치 처리 자동화 (EnemySystem.Update()에서 한 프레임에 모든 죽음 수집)
- **책임 분리**: Zone은 데미지, EnemySystem은 생명주기 관리
- **유지보수성**: 죽음 관련 버그 수정 시 EnemySystem 한 곳만 확인

### 12.2 성능 최적화 포인트

1. **EnemySystem.Update()**: 모든 적을 매 프레임 순회
   - 적이 많아지면 성능 저하 가능

## 15. 결론 및 설계 철학

### 15.1 핵심 설계 결정의 근거

이 아키텍처는 다음 원칙을 중심으로 설계되었습니다:

1. **명시성 > 암묵성**: 모든 게임 이벤트가 GameAction으로 명시적으로 표현됨
2. **조합 > 상속**: Effect와 TargetMode를 조합하여 복잡한 카드 생성
3. **데이터 중심**: ScriptableObject로 로직과 데이터 분리
4. **이벤트 기반**: 시스템 간 직접 의존성 최소화
5. **확장성 우선**: 새로운 기능을 기존 코드 수정 없이 추가 가능

### 15.2 이 아키텍처를 사용해야 하는 경우

- 턴 기반 또는 액션 기반 게임
- 복잡한 이벤트 체인이 있는 게임 (TCG, 로그라이크 등)
- 모딩 지원이 필요한 게임
- 리플레이 또는 네트워크 동기화가 필요한 게임

### 15.3 주의해야 할 경우

- 실시간 액션 게임 (ActionSystem의 오버헤드)
- 매우 단순한 게임 (과도한 추상화)
- 성능이 극도로 중요한 게임 (리플렉션 및 Boxing 비용)
